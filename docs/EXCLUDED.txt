===============================================================================
  EXCLUDED FROM V1 — What We Intentionally Did Not Build
===============================================================================

  This document explains what was deliberately left out of the v1
  prototype and why. These are conscious decisions, not oversights.

===============================================================================


1. PROTOCOL CREATION / EDITING
───────────────────────────────────────────────────────────────────────────────

  Excluded because:
    - Protocols are core Reportheld domain objects
    - Allowing external creation/modification risks data corruption
    - Template management is an admin function, not an automation need
    - External systems should trigger execution, not define what to execute


2. TEMPLATE MANAGEMENT
───────────────────────────────────────────────────────────────────────────────

  Excluded because:
    - Templates define the structure of protocols (fields, validations)
    - Exposing this externally blurs the boundary between API consumers
      and Reportheld administrators
    - No identified use case from automation consumers (n8n, AI)


3. USER AND GROUP MANAGEMENT
───────────────────────────────────────────────────────────────────────────────

  Excluded because:
    - High security risk (privilege escalation, tenant boundary violations)
    - Administrative operation that belongs in Reportheld's own UI
    - API consumers do not need to manage users


4. RAW DATABASE EXPOSURE / GENERIC CRUD
───────────────────────────────────────────────────────────────────────────────

  Excluded because:
    - The API is a "product surface", not a database mirror
    - CRUD endpoints (GET/PUT/DELETE on resources) would expose internal
      data structures and coupling to schema
    - Domain-driven actions (like "execute protocol") are safer and
      more meaningful than generic data operations


5. CROSS-SYSTEM DATA EXPORT
───────────────────────────────────────────────────────────────────────────────

  Excluded because:
    - Export is a complex, potentially large operation
    - Needs careful access control (what data can this tenant export?)
    - Better suited for a dedicated /api/v1/exports endpoint in v2
    - Read-only and lower risk, but scope is too large for prototype


6. GET ENDPOINTS FOR EXECUTION STATUS
───────────────────────────────────────────────────────────────────────────────

  Excluded because:
    - Webhooks already deliver real-time status updates
    - A GET endpoint would enable polling, which adds unnecessary load
    - May be added in v2 as a convenience/fallback
    - Design question: should it return full history or just current state?


7. WEBHOOK RETRY / DELIVERY GUARANTEES
───────────────────────────────────────────────────────────────────────────────

  Excluded because:
    - Adds significant complexity (queues, retry logic, dead letter)
    - Not critical for a prototype demonstrating the concept
    - Noted as a v2 improvement


8. RATE LIMITING
───────────────────────────────────────────────────────────────────────────────

  Excluded because:
    - Adds middleware complexity
    - Not needed in a prototype with 2 test API keys
    - Important for production; documented in IMPROVEMENTS.txt


9. AUTOMATED TESTS
───────────────────────────────────────────────────────────────────────────────

  Excluded because:
    - Prototype priority is proving the concept works
    - Manual verification via curl and test UI is sufficient
    - Test suite should be added before any production deployment


10. JWT / TOKEN-BASED AUTHENTICATION
───────────────────────────────────────────────────────────────────────────────

  Excluded because:
    - Adds complexity (token issuance, expiry, refresh)
    - Static API keys are sufficient for prototype
    - JWT is planned for v2 when the API goes beyond demo usage


===============================================================================
  END OF EXCLUSIONS
